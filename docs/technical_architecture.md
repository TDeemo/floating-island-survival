# 漂浮大陆：多平台技术架构方案

## 1. 项目概述

本方案针对《漂浮大陆》生存肉鸽游戏的多平台渐进开发路线进行技术架构设计。游戏核心玩法为“每日随机岛屿探险-资源收集-基地建造”循环，支持PC（Steam）优先，逐步扩展至移动端（iOS/Android）并实现全平台互通。

**核心设计约束**：
- 基于现有Unity项目资产（`temp/Assets/`）
- 1个月内交付可玩原型
- 支持动态昼夜循环、港口撤离压力机制
- 实现武器系统、岛屿生态环境差异化、种植系统、光照系统等复杂功能
- 确保多平台数据同步与账号互通

## 2. 技术栈选型

### 2.1 开发引擎与版本
| 组件 | 选型 | 理由 |
|------|------|------|
| 游戏引擎 | **Unity 2022.3 LTS** | 长期支持版本，2D工具链成熟，多平台构建支持完善，与现有资产兼容 |
| 渲染管线 | **Universal Render Pipeline (URP) 2D** | 提供高性能2D渲染，支持2D光照系统，移动端优化良好 |
| 物理引擎 | Unity内置2D Physics | 满足像素风游戏的碰撞检测需求，性能可控 |

### 2.2 核心插件与框架
| 插件 | 用途 | 备注 |
|------|------|------|
| **Unity Netcode for GameObjects (NGO)** | 实时网络同步（家园状态、种植生长、光照状态） | Unity官方方案，支持多平台，开发效率高 |
| **PlayFab SDK** | 后端服务（账号、数据存储、分析） | 免费层足够原型开发，提供完整的后端即服务 |
| **2D Light** (URP内置) | 实现光照系统 | 支持动态2D光源，可与Tilemap无缝集成 |
| **Cinemachine** | 2D相机控制 | 提供平滑跟随、镜头震动等效果 |
| **Unity UI Toolkit** | 高级UI系统 | 适合复杂UI（背包、建造界面），性能优于传统UGUI |
| **Git LFS** | 版本控制大文件 | 管理美术资源（像素图、动画） |

### 2.3 开发语言与工具
- **编程语言**：C# 9.0（Unity支持版本）
- **版本控制**：Git + Git LFS，分支策略采用Git Flow简化版
- **项目管理**：Unity DevOps（或Jira）+ GitHub Projects
- **持续集成**：GitHub Actions（自动化构建、部署测试包）

## 3. 系统架构设计

### 3.1 整体架构图（文字描述）

```
┌─────────────────────────────────────────────────────────────┐
│                       客户端 (Unity)                         │
├─────────────┬─────────────┬─────────────┬─────────────┤
│  核心游戏层 │    UI层     │  网络层     │  本地存储层 │
│  - 岛屿生成 │  - 背包/建造│  - NGO同步  │  - PlayerPrefs│
│  - 昼夜系统 │  - 种植界面 │  - PlayFab  │  - 缓存数据 │
│  - 战斗系统 │  - 光照控制 │  数据操作   │             │
│  - 种植系统 │             │             │             │
└─────────────┴─────────────┴─────────────┴─────────────┘
                               │
                     ┌─────────▼─────────┐
                     │    PlayFab云端    │
                     ├───────────────────┤
                     │  - 账号服务       │
                     │  - 玩家数据存储   │
                     │  - 岛屿种子库     │
                     │  - 成就与统计     │
                     └───────────────────┘
```

**数据流向**：
1. 玩家启动游戏 → PlayFab账号登录 → 拉取云端存档
2. 进入主岛 → 本地加载建筑布局，云端同步种植/生长状态
3. 开始探险 → 本地生成岛屿（种子来自云端），实时同步角色位置（NGO）
4. 资源收集 → 本地记录，撤离时批量同步至PlayFab
5. 建造/种植 → 立即本地生效，异步同步至云端

### 3.2 模块划分
1. **Gameplay Core**：岛屿生成、昼夜循环、战斗、采集、建造基础逻辑
2. **System Modules**：武器系统、种植系统、光照系统、天气系统
3. **UI Framework**：跨平台UI适配、响应式布局、输入反馈
4. **Network Layer**：NGO实时同步 + PlayFab REST API异步数据操作
5. **Data Management**：本地缓存、云端备份、数据版本控制
6. **Platform Adapter**：输入系统抽象、性能配置、商店接口封装

## 4. 后端服务架构

### 4.1 账号系统
- **方案**：PlayFab内置账号系统，支持Email/用户名登录、第三方（Steam、Apple、Google）OAuth集成
- **数据隔离**：每个账号对应唯一的PlayFab Player ID，作为所有数据的主键
- **安全性**：PlayFab提供客户端API密钥+服务器端Secret Key双层验证，敏感操作需服务器逻辑

### 4.2 数据存储结构
**Player Data (Key-Value存储)**：
| 数据键 | 数据类型 | 说明 |
|--------|----------|------|
| `island_seed_daily` | String | 每日岛屿种子（日期+随机偏移） |
| `buildings_layout` | JSON | 主岛建筑布局与状态 |
| `farming_plots` | JSON | 农田位置、作物类型、生长进度 |
| `inventory` | JSON | 背包物品列表（资源、武器、种子） |
| `research_progress` | JSON | 科技树解锁状态 |
| `player_stats` | JSON | 角色属性、成就、游戏时间等 |

**Global Data (只读共享)**：
| 数据键 | 用途 |
|--------|------|
| `island_templates` | 岛屿类型配置（生态、怪物、资源表） |
| `weapon_recipes` | 武器制作配方 |
| `crop_growth_data` | 作物生长参数（时间、产出） |

### 4.3 云存储策略
- **实时性要求高**：建筑放置、作物生长 → 使用NGO实时同步到所有在线客户端
- **实时性要求低**：背包物品、研究进度 → PlayFab UpdatePlayerData API异步保存（撤离时触发）
- **备份机制**：每日凌晨自动导出玩家数据到AWS S3（通过PlayFab CloudScript）

### 4.4 服务器逻辑（可选扩展）
**当前阶段**：完全依赖PlayFab无服务器架构，降低运维成本。
**未来扩展**：当需要复杂游戏逻辑（如PvP、交易市场）时，可部署自定义Azure Functions/ AWS Lambda，通过PlayFab CloudScript调用。

## 5. 数据同步与存储方案

### 5.1 同步模式
| 场景 | 同步方案 | 技术实现 |
|------|----------|----------|
| 主岛建筑放置/移动 | **实时同步** | NGO NetworkTransform + 自定义NetworkBehaviour |
| 作物生长进度 | **状态同步** | NGO NetworkVariable<float>，服务器权威更新 |
| 背包物品增减 | **批量异步** | PlayFab UpdatePlayerData，撤离时提交 |
| 岛屿生成种子 | **预同步** | 每日从PlayFab获取当日种子，确保所有玩家相同岛屿 |

### 5.2 冲突解决
- **乐观锁定**：PlayFab Data API支持版本号，提交时检查版本，冲突时客户端重新拉取
- **操作排队**：建造/种植操作在弱网环境下进入本地队列，恢复连接后顺序执行
- **数据压缩**：JSON数据使用MessagePack二进制序列化，减少传输量

### 5.3 离线支持
- 探险期间无网络 → 本地记录采集结果，返回主岛后检测网络并同步
- 建造/种植操作 → 本地立即显示效果，后台异步提交，失败时保留本地记录待重试

## 6. 多平台适配策略

### 6.1 输入系统抽象
**Unity Input System** 作为统一输入层：
- **PC**：键盘+鼠标 → 动作映射：WASD移动，鼠标瞄准/交互，快捷键1-9
- **移动端**：虚拟摇杆+触摸 → 动态UI控件，滑动手势，长按菜单
- **自适应逻辑**：运行时检测平台，动态切换输入映射

### 6.2 UI/UX适配
| 平台 | 布局策略 | 字体大小 | 交互优化 |
|------|----------|----------|----------|
| PC (1080p+) | 固定比例，侧边栏常驻 | 14-16pt | 鼠标悬停提示，右键菜单 |
| 移动端 (全面屏) | 响应式布局，底部操作栏 | 18-20pt | 触摸热区放大，防误触 |

**实现方案**：
- UI Toolkit Responsive Design：基于容器查询调整布局
- 平台专属UXML文件：PC版与移动版界面结构分离，资源按需加载

### 6.3 性能配置分级
| 平台 | 目标帧率 | 渲染分辨率 | 特效等级 | 岛屿生成范围 |
|------|----------|------------|----------|--------------|
| PC (中高配) | 60 FPS | 原生 | 高 | 大型岛屿(256×256) |
| PC (低配) | 30 FPS | 0.8x缩放 | 中 | 中型岛屿(128×128) |
| 移动端 (高端) | 30 FPS | 0.7x缩放 | 中 | 中型岛屿 |
| 移动端 (主流) | 30 FPS | 0.6x缩放 | 低 | 小型岛屿(64×64) |

**动态降级**：运行时检测设备性能指数，自动调整配置。

## 7. 核心系统技术实现

### 7.1 岛屿生成系统
**技术栈**：Unity Tilemap + Perlin噪声 + 脚本化对象(SO)

**实现要点**：
1. **分层生成**：
   - 基础地形层（Perlin噪声生成高度图）
   - 资源分布层（基于密度图随机放置节点）
   - 怪物巢穴层（难度权重控制）
   - 装饰层（环境物件，复用现有精灵）

2. **生态差异化**：
   - 每种岛屿类型对应一组SO配置（地形块、资源表、怪物池、天气参数）
   - 运行时根据岛屿类型切换配置

3. **性能优化**：
   - 分块加载：只生成玩家视野范围内的区块
   - 对象池：资源节点、敌人预制体复用

### 7.2 昼夜与港口撤离系统
**实现架构**：
- **TimeManager**单例：维护游戏内时间（24分钟现实时间 = 1游戏日）
- **DayNightCycle**组件：控制全局光照颜色、天空盒渐变
- **MonsterDifficultyScaler**：根据时间阶段动态调整怪物属性（刷新率、攻击力、追逐范围）
- **PortManager**：每座岛屿生成时计算港口位置，提供撤离触发器

**同步方案**：
- 游戏内时间由服务器权威（NGO NetworkVariable<float>），客户端插值平滑显示
- 怪物属性调整在服务端计算，客户端只读

### 7.3 武器系统
**数据驱动设计**：
- **WeaponScriptableObject**：定义武器属性（伤害、攻速、特效、获取方式）
- **WeaponManager**：管理玩家当前装备，处理切换、耐久消耗、特效播放
- **攻击逻辑**：复用现有 `AttackAnimationHandler.cs` 与 `DoudingAttack.cs`

**网络同步**：
- 武器切换：NGO NetworkVariable<int> 武器ID
- 耐久度：服务器权威计算，同步剩余耐久百分比

### 7.4 种植系统
**核心组件**：
1. **FarmPlot**：NetworkBehaviour，代表一块农田
   - 网络变量：`CropType`、`GrowthProgress` (0~1)、`WaterLevel`、`FertilizerLevel`
   - 客户端预测：种植/浇水操作立即本地反馈，服务器验证后正式更新

2. **CropDatabase**：ScriptableObject存储所有作物数据（生长时间、产出、季节限制）

3. **种子获取**：
   - 探险时采集野生植物：客户端随机掉落种子，服务器验证概率后写入PlayerData

**数据同步**：
- 生长进度：服务器每5分钟更新一次GrowthProgress，客户端平滑插值显示
- 收获操作：客户端发起，服务器扣除作物并添加物品到背包，广播给所有客户端

### 7.5 光照系统
**技术方案**：URP 2D Light + 自定义光照管理

**实现细节**：
1. **光源类型**：
   - **TorchLight**：附加到玩家角色的Point Light 2D，跟随移动
   - **CampfireLight**：静态Point Light 2D，放置后固定位置
   - **AmbientLight**：全局2D光源，模拟月光（夜晚降低强度）

2. **视觉效果**：
   - 光线衰减：调整Light的Falloff曲线，营造温馨感
   - 闪烁效果：使用Perlin噪声扰动光源强度（火把）
   - 战争迷雾：使用Tilemap阴影层，光源范围外设为不透明黑色

3. **网络同步**：
   - 火把点燃/熄灭：NGO RPC调用
   - 篝火状态：NetworkVariable<bool> isBurning
   - 光源参数（强度、范围）客户端自主，无需同步

4. **性能优化**：
   - 光源数量限制：移动端最多同时显示4个动态光源
   - 视锥剔除：屏幕外光源自动禁用渲染

### 7.6 天气系统
**组件化设计**：
- **WeatherProfile** SO：定义天气类型（雨、雪、雾）及其参数（粒子效果、声音、游戏影响）
- **WeatherManager**：根据岛屿类型与随机权重切换天气
- **粒子系统优化**：使用GPU Instancing渲染天气粒子，移动端降低粒子数量

## 8. 性能优化策略

### 8.1 渲染优化
- **2D批次合并**：确保Tilemap使用相同材质，减少Draw Calls
- **精灵图集**：将所有像素美术打包成少数几个4096×4096图集
- **动态分辨率**：移动端根据帧率动态调整渲染分辨率

### 8.2 内存管理
- **资源按需加载**：岛屿区块动态加载/卸载，使用Addressable Asset System
- **对象池**：敌人、资源节点、子弹等高频创建对象必须池化
- **纹理压缩**：移动端使用ASTC格式，PC端使用BC7

### 8.3 CPU优化
- **Job System + Burst Compiler**：岛屿生成算法、路径计算等密集型任务并行化
- **ECS轻量应用**：仅对大量同质实体（如粒子、装饰物）使用ECS架构
- **LOD系统**：远景Tilemap降低更新频率

### 8.4 网络优化
- **数据压缩**：使用MessagePack替代JSON，减少70%传输量
- **增量同步**：仅发送变化的数据字段
- **预测与调和**：移动、建造等操作客户端预测，服务器定期调和

## 10. 风险与应对

| 风险 | 可能性 | 影响 | 应对策略 |
|------|--------|------|----------|
| 多平台同步延迟 | 中 | 高 | 采用客户端预测+服务器调和，关键操作添加视觉补偿 |
| 移动端性能不足 | 高 | 中 | 提前建立性能配置分级，低端设备自动降级 |
| 后端扩展性瓶颈 | 低 | 高 | 使用PlayFab弹性架构，预留自定义服务器迁移路径 |
| 数据安全漏洞 | 中 | 高 | 服务器端验证所有关键操作，客户端仅做表现 |
| 苹果审核政策限制 | 中 | 中 | 提前研究iOS审核指南，避免使用私有API |
