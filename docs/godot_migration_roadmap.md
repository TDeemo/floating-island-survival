# 漂浮大陆游戏 Godot 迁移技术路线图

## 1. 项目背景与目标

### 1.1 迁移背景
由于 Unity Personal 许可证在容器化开发环境（headless）中无法激活，决策将《漂浮大陆》生存肉鸽游戏从 Unity 引擎迁移至 **Godot 4.5.1（Mono 版本）**。Godot 完全开源、无许可证激活障碍，确保长期构建稳定性。

### 1.2 迁移目标
- **技术目标**：在 6-8 周内产出 Godot 版本的可玩原型
- **质量目标**：保持原有游戏设计（8 大核心系统、像素风格、核心循环）不变，仅替换技术实现层
- **资产策略**：部分复用现有 Unity 像素美术纹理，其余引入 Godot 生态资源补充

### 1.3 时间线调整
原 Unity 路线图相应顺延：
- **可玩原型**：6-8 周（原 1 个月）
- **Alpha 测试**：第 12 周
- **Beta 测试**：第 24 周
- **最终上架**：第 36-48 周

## 2. 现有设计回顾（核心系统需求）

基于 `docs/game_design_document.md` 和 `docs/technical_architecture.md`，提取以下 10 个核心系统需求：

### 2.1 岛屿生成系统
- **功能**：程序化生成随机岛屿布局，基于 Perlin 噪声生成高度图
- **生态类型**：森林、矿山、沼泽、雪原、火山、丛林（各有特有怪物、资源、天气）
- **技术需求**：瓦片地图（TileMap）系统、分块加载、对象池优化
- **数据驱动**：岛屿类型配置（ScriptableObject → Godot Resource）

### 2.2 资源分布与采集系统
- **功能**：随机分布资源节点（木材、矿石、食物等），玩家采集交互
- **背包管理**：重量/格子限制，可升级扩容
- **工具系统**：不同工具对应不同资源采集效率（斧头→木材，镐→石头）

### 2.3 角色移动与控制系统
- **功能**：WASD/虚拟摇杆控制角色移动，2D 相机跟随
- **动画状态机**：行走、攻击、采集、空闲等动画切换
- **平台适配**：PC（键鼠）与移动端（触控）输入抽象

### 2.4 武器系统
- **获取方式**：制作、怪物掉落、宝箱掉落、任务奖励
- **属性**：伤害、攻速、范围、耐久度、特殊效果（击退、中毒、冰冻、燃烧、穿透）
- **装备限制**：主武器+副武器，负重影响移动速度

### 2.5 怪物 AI 系统
- **行为树**：巡逻、追击、攻击、返回等状态
- **动态难度**：随游戏内时间（昼夜）变化怪物属性（刷新率、攻击力、追逐范围）
- **敌人类型**：近战、远程、飞行、BOSS（各有不同攻击模式）

### 2.6 昼夜循环与港口撤离系统
- **时间管理**：游戏内时间（24 分钟现实时间 = 1 游戏日），白天 18 分钟，夜晚 6 分钟
- **港口生成**：每座岛屿边缘固定生成港口区域，提供撤离交互
- **紧张感机制**：夜晚怪物能力大幅提升，玩家必须在夜晚前抵达港口撤离

### 2.7 建造系统
- **功能**：在主岛放置建筑（工作台、熔炉、厨房、仓库、防御塔、研究所）
- **建造流程**：进入建造模式、选择建筑、放置、扣除资源
- **升级路径**：建筑可多次升级，提升效率、容量或解锁新配方

### 2.8 种植系统
- **核心概念**：农田种植从探险岛屿采集的种子，作物用于食用或制作治疗道具
- **种子获取**：只能通过探险采集野生植物获得，无法通过种植自循环
- **生长因素**：天气、季节、肥料影响生长速度与产量

### 2.9 光照系统
- **光源类型**：手持火把（移动点光源）、篝火（静态点光源）、环境光（月光）
- **游戏机制**：黑暗区域完全不可见（战争迷雾），光源吸引怪物但提供视野
- **视觉效果**：光线衰减、闪烁效果、动态混合

### 2.10 天气系统
- **天气类型**：雨、雪、雾、毒雾、暴雪、高温等
- **影响机制**：影响视野、移动速度、资源再生、玩家状态
- **季节周期**：每 30 天游戏内时间为一个季节，资源分布变化

## 3. Godot 技术栈选型

### 3.1 引擎与版本
- **引擎**：Godot Engine 4.5.1（稳定版）
- **版本**：Mono 版本（支持 C# 备选，但主推 GDScript）
- **渲染器**：Forward+（默认），支持 2D 光照系统
- **目标平台**：Windows/Linux（PC 原型）→ 后续扩展 Android/iOS

### 3.2 编程语言策略
- **主语言**：GDScript（Godot 原生，类似 Python，学习曲线平缓）
- **备选语言**：C#（仅用于性能关键算法，如岛屿生成噪声计算）
- **脚本模板**：统一提供基础脚本模板（含文档字符串、类型提示）

### 3.3 项目结构规范
```
godot_project/
├── assets/                  # 所有美术资源
│   ├── sprites/            # 精灵纹理（复用 Unity PNG）
│   ├── tilesets/           # 瓦片集配置
│   ├── backgrounds/        # 环境背景
│   ├── ui/                 # UI 元素
│   └── audio/              # 音效与音乐
├── scenes/                 # 场景文件
│   ├── levels/             # 岛屿场景
│   ├── objects/            # 可交互对象（资源节点、建筑等）
│   ├── characters/         # 角色预制场景
│   └── ui/                 # UI 场景
├── scripts/                # GDScript 代码
│   ├── core/               # 核心系统
│   │   ├── island_generator.gd
│   │   ├── time_manager.gd
```

### 3.4 工具链配置
- **版本控制**：Git + .gitignore（排除 `.import/`、`*.translation` 等）
- **编辑器配置**：统一编辑器设置（缩进、自动换行、代码完成）
- **导入预设**：为像素艺术纹理创建导入预设（过滤关闭、像素完美）
- **构建脚本**：使用 Godot CLI 进行自动化构建

### 3.5 版本控制策略
- **分支模型**：简化 Git Flow
  - `main`：稳定版本，对应里程碑
  - `develop`：开发集成分支
  - `feature/*`：功能分支（如 `feature/island-generation`）
  - `hotfix/*`：紧急修复
- **提交规范**：约定式提交（Conventional Commits）
- **代码审查**：PR 模板，至少一人审查

## 4. GDScript 编码规范

### 4.1 命名约定
- **文件命名**：小写蛇形（`island_generator.gd`）
- **类/节点命名**：大驼峰（`IslandGenerator`）
- **变量/函数**：小写蛇形（`current_time`、`calculate_damage()`）
- **常量**：大写蛇形（`MAX_INVENTORY_SIZE`）
- **私有成员**：前缀下划线（`_internal_cache`）

### 4.2 文件组织
- 每个主要类/系统单独文件
- 工具函数集中到 `utils/` 目录
- 资源定义使用 Godot Resource 而非硬编码

### 4.3 代码风格
```gdscript
# 文档字符串（使用三引号）
## 岛屿生成器主类
class_name IslandGenerator
extends Node2D

# 常量定义
const TILE_SIZE := 16
const MAX_ISLAND_WIDTH := 256

# 导出变量（带类型提示）
@export var noise_seed: int = 12345
@export var island_types: Array[IslandTypeResource]

# 私有变量
var _noise := FastNoiseLite.new()
var _generated_chunks := {}

# 信号定义
signal island_generated(island_data: Dictionary)

# 公共函数（带类型提示）
## 生成指定种子和大小的岛屿
func generate_island(seed: int, width: int, height: int) -> TileMap:
    _setup_noise(seed)
    var tilemap := _create_base_tilemap(width, height)
    _generate_resources(tilemap)
    island_generated.emit({"seed": seed, "width": width, "height": height})
    return tilemap

# 私有函数
func _setup_noise(seed: int) -> void:
    _noise.seed = seed
    _noise.frequency = 0.05
```

### 4.4 最佳实践
- 使用 `@export` 暴露可调参数而非硬编码
- 利用 Godot 信号系统进行解耦
- 优先使用 Resource 进行数据驱动设计
- 为性能关键路径考虑 C# 实现

## 5. 开发路线图（6-8周详细计划）

### 第1-2周：资产迁移与基础搭建
**目标**：完成可复用纹理导入，搭建基础项目结构，实现主岛场景与角色移动

**具体任务**：
1. **资产复制与纹理导入**（基于任务34报告）
   - 复制205个PNG纹理到 `assets/sprites/`
   - 创建像素艺术导入预设（过滤关闭、压缩优化）
   - 配置瓦片集（TileSet）使用 `Tilemap_Elevation.png` 和 `Tilemap_Flat.png`

2. **基础场景创建**
   - 主岛场景（`scenes/levels/main_island.tscn`）：配置地形、天空盒、基础光照
   - 角色场景（`scenes/characters/player.tscn`）：Sprite2D、CollisionShape2D、动画状态机
   - 相机系统：Cinemachine2D或自定义相机跟随脚本

3. **角色移动与输入系统**
   - GDScript实现WASD/虚拟摇杆移动控制
   - 输入抽象层（支持PC键鼠与移动触控切换）
   - 基础动画状态机（行走、空闲）

4. **基础UI框架**
   - 主菜单场景
   - HUD基础元素（时间显示、背包图标）

**交付物**：
- 可运行的Godot项目（角色可在主岛移动）
- 资产导入报告与配置文档

### 第3-4周：核心系统迁移（第一波）
**目标**：实现岛屿生成、资源分布、基础采集系统

**具体任务**：
1. **岛屿生成系统**
   - 实现Perlin噪声地形生成算法（GDScript或C#）
   - 支持多种生态类型（森林、矿山、沼泽等）
   - 瓦片地图动态加载与对象池优化

2. **资源分布系统**
   - 资源节点场景（木材、矿石、草药等）
   - 采集交互逻辑（工具检测、动画反馈）
   - 背包系统基础实现（物品槽、重量计算）

3. **基础探险循环**
   - 场景切换：主岛 → 随机岛屿
   - 基础时间系统（游戏内时间显示）
   - 简单港口撤离机制

4. **测试与调试**
   - 岛屿生成稳定性测试
   - 采集交互功能验证

**交付物**：
- 可随机生成岛屿并采集资源的原型
- 岛屿生成算法性能报告

### 第5-6周：核心系统迁移（第二波）
**目标**：实现武器系统、怪物AI、昼夜循环

**具体任务**：
1. **武器系统**
   - 武器数据资源定义（ScriptableObject → Godot Resource）
   - 武器获取逻辑（制作、掉落、宝箱）
   - 攻击系统集成（动画、伤害计算、特效）

2. **怪物AI系统**
   - 敌人行为树实现（巡逻、追击、攻击）
   - 动态难度调整（基于昼夜时间）
   - BOSS战斗逻辑

3. **昼夜循环系统**
   - 完整时间管理系统（24分钟现实时间 = 1游戏日）
   - 视觉效果（天色渐变、光照调整）
   - 港口撤离压力机制（夜晚怪物增强）

4. **战斗测试**
   - 武器与敌人平衡性初步调整
   - 昼夜难度曲线验证

**交付物**：
- 包含战斗和昼夜系统的可玩原型
- 武器与敌人数值平衡表

### 第7-8周：系统集成与原型打磨
**目标**：集成建造、种植、光照系统，产出完整可玩原型

**具体任务**：
1. **建造系统**
   - 建造模式UI实现
   - 建筑放置与碰撞检测
   - 建筑升级逻辑

2. **种植系统**
   - 农田场景与交互
   - 作物生长状态机（天气、季节影响）
   - 种子获取与使用逻辑

3. **光照系统**
   - Light2D配置（火把、篝火、环境光）
   - 战争迷雾效果实现
   - 性能优化（光源数量限制）

4. **原型集成测试**
   - 完整核心循环验证（探险→采集→建造→种植）
   - 多系统交互调试
   - 性能分析与优化

**交付物**：
- **完整可玩PC原型**（包含所有8个核心系统）
- 用户测试反馈报告
- 性能分析文档

## 6. 风险识别与应对策略

### 6.1 技术风险
| 风险 | 可能性 | 影响 | 应对策略 |
|------|--------|------|----------|
| **Godot 2D光照性能不足** | 中 | 高 | 1. 早期性能测试 2. 光源数量限制 3. 备选简化方案（着色器模拟） |
| **岛屿生成算法效率低** | 高 | 中 | 1. 使用C#实现核心算法 2. 分块异步生成 3. 预生成+缓存 |
| **GDScript调试困难** | 低 | 中 | 1. 单元测试覆盖 2. 使用Godot内置调试器 3. 日志系统 |
| **像素风格一致性保持** | 中 | 中 | 1. 统一导入预设 2. 风格参考板 3. 手动调整新资源 |

### 6.2 进度风险
| 风险 | 可能性 | 影响 | 应对策略 |
|------|--------|------|----------|
| **动画转换耗时超预期** | 高 | 中 | 1. 优先核心动画（行走、攻击） 2. 简化次要动画 3. 使用Godot动画工具优化流程 |
| **系统集成复杂度高** | 中 | 高 | 1. 模块化设计 2. 接口先行 3. 持续集成测试 |
| **新引擎学习曲线** | 低 | 低 | 1. 提供GDScript模板 2. 代码审查指导 3. 知识共享文档 |

### 6.3 兼容性风险
| 风险 | 可能性 | 影响 | 应对策略 |
|------|--------|------|----------|
| **纹理导入失真** | 中 | 低 | 1. 批量测试脚本 2. 人工检查关键纹理 3. 备份原始文件 |
| **输入系统跨平台问题** | 低 | 中 | 1. 抽象输入层 2. 多设备测试 3. 用户配置选项 |
